# M0: 骨架搭建 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 建立一个可运行的 pnpm monorepo：三个可发布空包（core / table / theme）能被 tsdown 打包，playground 可 dev、docs 可访问，CI 三步（lint / typecheck / build）全绿。

**Architecture:** pnpm workspace + turbo-lite（用根 scripts 而非引入 turborepo，避免多学一层）。每个包各自的 tsdown.config、tsconfig extends 共享 base，Oxlint + Prettier 全局配置。CI 通过 GitHub Actions 跑 install → lint → typecheck → build。

**Tech Stack:** pnpm 9+, TypeScript 5.6+, tsdown, Vite 8, VitePress, Oxlint, Prettier, Changesets, Vitest（M0 只做 setup，不写业务测试）, GitHub Actions

**Spec reference:** [`../specs/2026-07-01-draggable-table-design.md`](../specs/2026-07-01-draggable-table-design.md) §19 M0

## File Structure

```
draggable-table/
├── package.json                           根 workspace + scripts
├── pnpm-workspace.yaml                    workspace globs
├── tsconfig.base.json                     共享 tsconfig
├── .oxlintrc.json                         Oxlint 配置
├── .prettierrc                            Prettier 配置
├── .prettierignore
├── .gitignore                             忽略 node_modules / dist
├── .npmrc                                 pnpm 配置（strict-peer-deps 等）
├── .changeset/config.json                 Changesets 配置
├── .github/workflows/ci.yml               三步 CI
├── packages/
│   ├── core/
│   │   ├── package.json                   @draggable-table/core
│   │   ├── tsconfig.json                  extends base
│   │   ├── tsdown.config.ts
│   │   └── src/index.ts                   export {} 空实现
│   ├── table/
│   │   ├── package.json                   @draggable-table/table
│   │   ├── tsconfig.json
│   │   ├── tsdown.config.ts
│   │   └── src/index.tsx                  export {} 空实现
│   └── theme/
│       ├── package.json                   @draggable-table/theme
│       └── src/index.css                  空 CSS
├── apps/
│   ├── playground/
│   │   ├── package.json
│   │   ├── tsconfig.json
│   │   ├── vite.config.ts
│   │   ├── index.html
│   │   └── src/main.tsx                   显示 "Hello Draggable Table"
│   └── docs/
│       ├── package.json
│       ├── .vitepress/config.ts
│       └── index.md
└── tests/e2e/                             Playwright 目录（M0 只建目录）
    └── .gitkeep
```

**责任划分：** 每个 `package.json` 只声明自己的 deps + exports；`tsconfig.base.json` 是唯一的类型规则源；`.oxlintrc.json` 是唯一的 lint 规则源；每个包 `tsdown.config.ts` 只描述该包的入口与产物格式。playground 和 docs 是消费方——它们通过 workspace 协议引用 packages/*。

## Deviations from plan during execution

以下改动是 M0 执行时相对本 plan 的实际偏离，按主题分组。规划下一阶段（M1）时以本节为准，不要以 plan 正文的原始条目为准。

### 1. tsdown 版本与产物扩展名

- **plan**：`tsdown: ^0.6.0`，产物 `index.js` / `index.cjs` / `index.d.ts`
- **实际**：`tsdown: ^0.22.0`。0.6 系列与当前 `rolldown` API 已不兼容（`transformPlugin` 导出不存在）。0.22 默认产物扩展名为 `.mjs` / `.cjs` / `.d.mts` / `.d.cts`。
- **连带修改**：`packages/core/package.json` 和 `packages/table/package.json` 的 `main` / `module` / `types` / `exports` 全部改成新扩展名，仍是 ESM + CJS + d.ts 三形态，语义不变。

### 2. 放弃 TS project references / composite，改用 paths 别名

- **plan**：每个 package 的 tsconfig 声明 `composite: true`，消费方 tsconfig 用 `references` 指向依赖包；root tsconfig 用 solution style 汇总
- **实际**：所有 tsconfig 不再有 `composite` 和 `references`。消费方（table、playground）用 `compilerOptions.paths` 直接映射到同级包的 `src/` 源文件。
- **理由**：
  - `tsc -b` 我们完全不用（build 由 tsdown 承担），composite 模式失去价值
  - `composite: true` + `references` 会触发 TS6305（referenced project 的 `.d.ts` 必须已产出）——但 tsdown 产出的是 `.d.mts` / `.d.cts`，TS 默认查找的 `.d.ts` 不存在，`pnpm typecheck` 在刚 clone 的仓库上必然失败
  - paths 直指源文件后，`typecheck` 完全独立于 `build`，CI 顺序 lint → typecheck → build 才成立
- **代价**：typecheck 时消费方会把依赖包的 src 也纳入编译单元，未来单包源码量大后 typecheck 会变慢。如果实际观测到明显变慢，届时再迁回 composite，并把 CI 顺序改成 build → typecheck。
- **注意**：paths（typecheck 用）和 `dependencies: workspace:*`（bundling 用）是两条独立链路。core 新增子模块导出时两处都要同步更新。

### 3. Root tsconfig 从 solution index 改为 orphan config 文件的 catch-all

- **plan**：`tsconfig.json` 是 solution style（`files: []` + `references`）
- **实际**：`tsconfig.json` extends base，`include` 只覆盖 `*.config.ts` / `packages/*/*.config.ts` / `apps/*/*.config.ts`。加装 `@types/node` 让 `vite.config.ts` / `tsdown.config.ts` 引用的 `node:*` 内置模块有类型。
- **理由**：包根目录下的 `vite.config.ts` / `tsdown.config.ts` 不在任何子 tsconfig 的 `include` 内，编辑器会 fallback 到 TS 默认设置，`skipLibCheck` 关着，Vite/tsdown 的类型往下走 `node:http` 等报红。
- **连带修改**：根 `package.json` 的 `typecheck` script 前置 `tsc --noEmit -p .` 让 CI 也检查这些文件。

### 4. dnd-kit / plugin-react 版本对齐当前 npm

- `@dnd-kit/sortable`：^8.0.0 → **^10.0.0**（plan 版本已落后 2 个 major）
- `@dnd-kit/modifiers`：^7.0.0 → **^9.0.0**
- `@vitejs/plugin-react`：^5.0.0 → **^6.0.0**
- `@dnd-kit/core`：^6.1.0 保持不变（当前 npm 6.3.1 在 caret 范围内）

### 5. Windows 兼容修补

- 根 `package.json` 的 `build` / `build:apps` script 里 `pnpm --filter './packages/*'` 单引号在 cmd.exe 下被当作字面量，改为转义双引号 `\"./packages/*\"`
- `.gitignore` 的 `.vitepress/cache/` 因为路径中有 `/` 被 git 锚定到根，加 `**/` 前缀改为 `**/.vitepress/cache/` 才能匹配 `apps/docs/.vitepress/cache/`
- `.gitignore` 添加 `*.tsbuildinfo`

### 6. theme 包补类型入口

- 增加 `packages/theme/src/index.d.ts`（一行 `export {}`），`package.json` 加 `types` 字段和 `exports.".".types` 条件
- 消费方 `import '@draggable-table/theme'` 时不再触发 VSCode 的 "cannot find module" 红字
- 这个文件是 M0 收尾时补的后置补丁，M2 如果 theme 引入真正的 TS token 定义，届时应把它纳入某个工程的 include

### 7. TS 6.0 弃用清理

- 所有 tsconfig 删除 `baseUrl`（TS 6 标记弃用，7.0 移除）。TS 4.1+ 起 `paths` 相对键值以 tsconfig 所在目录为基准解析，`baseUrl` 已无必要。

### 8. 提交侧工作流工具（超出 M0 plan 范围，作为收尾补齐）

Plan 未提及但常规工程实践缺失，M0 收尾一并补上：

- **husky v9** + `.husky/pre-commit`（触发 lint-staged）+ `.husky/commit-msg`（触发 commitlint）
- **lint-staged**：pre-commit 只对 staged 文件跑 prettier + oxlint
- **commitlint**：`extends @commitlint/config-conventional`，强制 Conventional Commits
